# Directory Guide

#### 언제든 바뀔 수 있음. 그냥 권장 예시일 뿐, 훨씬 좋은 디렉토리 구조가 있다면 refactor/* 브랜치에 넣으면 됨.

## 목적

이 문서는 정명 프로젝트의 디렉토리 구조와 파일 배치 기준을 정리하기 위해 작성한다.  
파일을 새로 만들거나 구조를 수정할 때, 어떤 책임을 어디에 둘지 일관되게 판단할 수 있도록 하는 것을 목표로 한다.

## 핵심 원칙

- 디렉토리는 기술 자체보다 역할과 책임을 기준으로 나눈다.
- 하나의 디렉토리는 가능한 한 하나의 주요 책임을 가진다.
- 공통으로 재사용되는 코드는 명확한 공용 영역으로 분리한다.
- 파일은 사용하는 위치와 가장 가까운 곳에 둔다.
- 구조 변경은 단순 이동이 아니라 책임 분리를 위한 변경이어야 한다.

## 최상위 디렉토리

### `front/`
사용자에게 보이는 클라이언트 애플리케이션을 담당한다.

포함 예시:
- 화면
- UI 컴포넌트
- 클라이언트 상태관리
- 라우팅
- 사용자 입력 처리
- API 호출

### `back/`
서버 애플리케이션을 담당한다.

포함 예시:
- API 엔드포인트
- 비즈니스 로직
- 인증/인가
- 데이터베이스 접근
- 도메인 규칙
- 서버 설정

### `docs/`
프로젝트 진행에 필요한 문서를 관리한다.

포함 예시:
- Git 전략 문서
- 디렉토리 구조 문서
- API 규칙 문서
- MVP 범위 문서
- 배포 문서

## Frontend 디렉토리 가이드

`front/src` 내부는 아래 원칙을 기준으로 구성한다.

### `app/` 또는 `pages/`
라우팅과 화면 진입 지점을 둔다.

포함 예시:
- 페이지 컴포넌트
- 라우트 단위 파일
- 레이아웃

### `components/`
여러 곳에서 재사용 가능한 UI 컴포넌트를 둔다.

포함 예시:
- 버튼
- 입력창
- 모달
- 카드
- 헤더
- 공통 레이아웃 요소

규칙:
- 특정 기능에 강하게 종속되지 않은 UI만 둔다.
- 기능 전용 컴포넌트는 `features/` 쪽에 둔다.

### `features/`
기능 단위로 묶이는 프론트 로직을 둔다.

포함 예시:
- 로그인 기능
- 토론 생성 기능
- 토론 스레드 기능
- 합의 생성 기능

규칙:
- 기능과 관련된 UI, 훅, 상태, API 호출 코드를 함께 둘 수 있다.
- “이건 어떤 기능에 속하는가?”가 분명하면 `features/`를 우선 고려한다.

### `shared/`
여러 기능에서 함께 사용하는 공통 코드를 둔다.

포함 예시:
- 공통 타입
- 상수
- 유틸 함수
- 공통 API 클라이언트
- 공통 훅

규칙:
- 여러 곳에서 실제로 재사용될 때만 둔다.
- 처음부터 무조건 공용으로 빼지 않는다.

### `assets/`
정적 리소스를 둔다.

포함 예시:
- 이미지
- 아이콘
- 폰트
- 정적 파일

## Backend 디렉토리 가이드

`back/src` 내부는 아래 원칙을 기준으로 구성한다.

### `modules/`
기능 단위의 서버 모듈을 둔다.

포함 예시:
- auth
- debate
- thread
- consensus
- report

규칙:
- 가능하면 기능 단위로 묶는다.
- 관련된 controller, service, repository, dto를 함께 관리한다.

예시:
- `modules/auth`
- `modules/debate`
- `modules/consensus`

### `common/`
여러 모듈에서 함께 사용하는 공통 서버 코드를 둔다.

포함 예시:
- 예외 처리
- 공통 응답 형식
- 가드
- 인터셉터
- 데코레이터
- 유틸 함수

### `database/`
데이터베이스 관련 설정과 공통 접근 구성을 둔다.

포함 예시:
- DB 설정
- 커넥션 설정
- 마이그레이션 관련 파일
- 시드 데이터

### `domain/`
프로젝트의 핵심 도메인 개념과 규칙을 별도로 관리해야 할 때 사용한다.

포함 예시:
- 토론
- 용어 정의
- 합의
- 사용자 권한 규칙

규칙:
- 단순 CRUD를 넘어서 핵심 규칙이 커질 때 분리한다.
- 초기에는 `modules/` 중심으로 가고, 도메인 복잡도가 높아질 때 강화한다.

## 파일 배치 기준

새 파일을 만들 때는 아래 순서로 판단한다.

1. 이 파일은 어느 기능에 속하는가?
2. 특정 기능 전용인가, 여러 곳에서 공통으로 쓰이는가?
3. UI인가, 서버 로직인가, 문서인가?
4. 현재 디렉토리에 두는 것이 책임상 자연스러운가?

판단 기준:
- 기능 전용이면 기능 디렉토리에 둔다.
- 여러 기능이 공유하면 공통 디렉토리에 둔다.
- 애매하면 더 좁은 범위에 두고, 실제 재사용이 확인되면 분리한다.

## 구조 설계 원칙

### 1. 공통화는 늦게 한다
아직 한 곳에서만 쓰는 코드는 섣불리 `shared/`, `common/`으로 빼지 않는다.  
중복이 실제로 확인된 뒤 분리한다.

### 2. 기능 응집도를 우선한다
서로 같이 수정되는 파일들은 가능한 한 가까이 둔다.  
파일 종류보다 작업 단위를 기준으로 묶는 것을 우선한다.

### 3. 이름은 역할이 드러나게 짓는다
디렉토리와 파일 이름만 보고도 대략적인 책임을 알 수 있어야 한다.

좋은 예시:
- `auth-service.ts`
- `debate-controller.ts`
- `consensus-form.tsx`

피해야 할 예시:
- `util2.ts`
- `temp.ts`
- `commonThing.ts`

### 4. 깊이를 과도하게 늘리지 않는다
디렉토리 구조는 지나치게 깊어지면 찾기 어려워진다.  
초기 MVP 단계에서는 2~3단계 정도의 탐색 깊이를 우선한다.

## 문서 관리 원칙

문서는 `docs/`에서 관리한다.

예시:
- `docs/gitflow.md`
- `docs/directory-guide.md`
